You are building a lightweight, production-oriented multimodal document comparison system.

---

## 🎯 Objective

Compare two documents:

* One in raw text format
* One in image format

Determine how similar they are in meaning, structure, and key information.

This system should simulate real-world enterprise use cases such as:

* Invoice validation
* Document deduplication
* Ticket attachment verification

---

##  Key Design Constraint

This system must be **lightweight, interpretable, and scalable**.

DO NOT use heavy multimodal transformers.

Instead, implement a **hybrid approach** combining:

* OCR
* Semantic embeddings
* Entity extraction
* Lightweight layout-aware heuristics

---

##  Required Pipeline Architecture

Structure the system into clearly defined modules:

1. Input Handling
2. OCR Extraction
3. Text Preprocessing
4. Feature Extraction
5. Multi-level Similarity Computation
6. Score Aggregation
7. Explainability Output

Each module must be implemented as a separate function.

---

##  Step 1: Input Handling

* Accept:

  * Raw text input
  * Image file input

* Validate inputs and handle edge cases

---

##  Step 2: OCR (Image → Text)

* Use a lightweight OCR method (e.g., pytesseract)

* Extract:

  * Raw text
  * Line-level structure if possible

* In code comments and README, explicitly explain:

  * OCR limitations (noise, missing structure, misreads)
  * How these affect similarity scoring

---

##  Step 3: Text Preprocessing

Apply consistent preprocessing to BOTH:

* Original text
* OCR-extracted text

Include:

* Normalization
* Noise removal (controlled — avoid losing key info)

Explain trade-offs:

* Cleaning vs losing important tokens (numbers, IDs, etc.)

---

##  Step 4: Feature Engineering (Important)

Extract multiple representations:

### A. Semantic Representation

* Generate sentence/document embeddings
* Use a lightweight model (e.g., MiniLM)

---

### B. Lexical Representation

* TF-IDF or token-based representation

---

### C. Entity Extraction (Critical)

Extract structured information:

* Dates
* Numbers / amounts
* Names / identifiers

Explain:

* Why entity matching is crucial in enterprise workflows

---

### D. Lightweight Layout Heuristics (Key Requirement)

Simulate layout awareness WITHOUT heavy models.

Extract simple features such as:

* Line index (position in document)
* Relative position (top / middle / bottom)
* Grouping of nearby lines

Use these to:

* Improve matching of semantically similar but differently ordered text

Explain clearly:

* Why layout matters
* How this approximates layout-aware models

---

##  Step 5: Multi-Level Similarity

Implement ALL of the following:

---

### 1. Lexical Similarity

* TF-IDF + cosine similarity OR Jaccard similarity

Explain:

* Why this is brittle (fails on paraphrasing)

---

### 2. Semantic Similarity (Core)

* Embedding-based cosine similarity

Explain:

* Why this captures meaning better than lexical methods

---

### 3. Entity-Level Similarity

* Compare extracted entities
* Measure overlap / consistency

Explain:

* Why this is critical for correctness (e.g., invoices)

---

### 4. Layout-Aware Adjustment (Lightweight)

Incorporate layout heuristics into similarity:

* Penalize mismatches in key sections
* Reward aligned structure

---

##  Step 6: Final Score

Combine all similarity scores into a final score.

You MUST:

* Define weights (e.g., semantic > entity > lexical)
* Justify weights in:

  * Code comments
  * README

Example (do not hardcode blindly — justify):
final_score = weighted combination of all similarity signals

---

##  Step 7: Explainability Output (Very Important)

Return not just a score, but:

* Individual component scores
* Key matched entities
* Mismatched entities
* Sections that differ significantly

This is critical for real-world debugging and trust.
